## Подробный отчет

## Назначение отчета

Этот отчет описывает практическую работу с Linux и сетями в рамках двух учебных проектов: базовое администрирование Linux и настройка сетевой инфраструктуры на виртуальных машинах. Текст подготовлен для портфолио и резюме: он делает акцент не на формальных условиях заданий, а на том, какие инженерные действия были выполнены, какие инструменты были освоены и какой практический опыт был получен.

Работа выполнялась в среде виртуализации на Ubuntu Server 24.04 LTS без графического интерфейса. Такой формат близок к реальной серверной эксплуатации: основная работа велась через терминал, конфигурационные файлы, системные службы, сетевые утилиты и анализ логов.

## Краткое резюме проделанной работы

В ходе двух проектов была последовательно пройдена цепочка от установки и первичной настройки Linux-сервера до построения многоузловой сетевой схемы с маршрутизацией, firewall-правилами, DHCP и NAT.

Были выполнены следующие направления работ:

- установка и базовая настройка Ubuntu Server в VirtualBox;
- создание пользователей, работа с группами и правами sudo;
- настройка hostname, часового пояса, синхронизации времени и сетевых параметров;
- переход от DHCP к статической конфигурации сети через netplan;
- установка, запуск и настройка SSH-сервера на нестандартном порту;
- диагностика процессов, памяти, CPU, дисков, файловых систем и системных журналов;
- работа с cron и проверка выполнения заданий через логи;
- расчет IPv4-сетей, масок, диапазонов хостов, частных и публичных адресов;
- ручная и постоянная статическая маршрутизация между виртуальными машинами;
- измерение пропускной способности соединений через iperf3;
- настройка iptables и проверка состояния хостов через ping и nmap;
- построение сети из рабочих станций и маршрутизаторов;
- включение IP forwarding и анализ маршрутов;
- диагностика маршрута через traceroute/tcpdump и ICMP;
- настройка DHCP-сервера, выдача адресов и привязка адреса к MAC;
- настройка NAT, SNAT/DNAT и проверка TCP-доступности через telnet.

Результатом стала не только серия выполненных конфигураций, но и понимание связей между уровнями системы: как пользовательские команды меняют файлы конфигурации, как systemd применяет настройки служб, как ядро Linux использует таблицу маршрутизации, как firewall влияет на ICMP/TCP-трафик, и как логи помогают подтверждать состояние системы.

## Среда и подход к работе

Практика выполнялась на виртуальных машинах с Ubuntu Server 24.04 LTS. Использование серверной версии без GUI позволило сфокусироваться на командной строке и реальных механизмах администрирования: пакетном менеджере, systemd, конфигурационных файлах, сетевых интерфейсах, службах и логах.

Для изоляции и безопасных экспериментов использовалась виртуализация. Это позволило поднимать несколько независимых машин, менять сетевые схемы, проверять маршрутизацию, ломать и восстанавливать доступность, не затрагивая основную систему.

Работа велась итерационно:

1. Изучить текущее состояние системы или сети.
2. Внести минимально необходимое изменение в конфигурацию.
3. Применить изменение через системную утилиту или перезапуск службы.
4. Проверить результат диагностическими командами.
5. Зафиксировать вывод, скриншот или объяснение причины результата.

Такой подход помог выработать привычку не просто выполнять команду, а подтверждать ее эффект: через `ip a`, `ip r`, `systemctl`, `journal`/логи, `ping`, `netstat`, `top`, `htop`, `df`, `du`, `tcpdump`, `nmap`, `telnet` и другие инструменты.

## Часть 1. Базовое администрирование Linux

### Установка и первичная настройка сервера

Работа началась с установки Ubuntu Server 24.04 LTS без графической оболочки. Это дало практический опыт подготовки минимальной серверной среды, где все управление выполняется через терминал. После установки была проверена версия системы через системный файл с информацией о релизе.

В процессе первичной настройки были выполнены базовые действия, типичные для подготовки сервера:

- проверка версии установленной ОС;
- настройка имени хоста;
- установка часового пояса;
- проверка доступных сетевых интерфейсов;
- определение внутреннего IP-адреса, шлюза и внешнего адреса;
- проверка сетевой связности после изменения конфигурации.

Главный практический результат этой части: появилось понимание, что серверная настройка Linux состоит из последовательных проверяемых шагов. Недостаточно изменить параметр; важно убедиться, что он переживает перезагрузку и действительно используется системой.

### Пользователи, группы и sudo

Была выполнена работа с локальными учетными записями: создан дополнительный пользователь, проверено его наличие в системной базе пользователей, выполнено добавление в административные группы.

Отдельно была освоена логика `sudo`: выполнение административных действий не из-под root напрямую, а через пользователя с делегированными правами. На практике это применялось при изменении hostname от имени созданного пользователя.

Практическая ценность этого блока:

- понимание различий между обычным пользователем, root и пользователем с sudo-доступом;
- знакомство с группами как механизмом выдачи прав;
- проверка результата через системные файлы и команды;
- понимание, почему в серверной эксплуатации прямой вход под root часто заменяется управляемым sudo-доступом.

### Сетевая конфигурация через netplan

Один из ключевых блоков базового проекта был связан с настройкой сети. Сначала машина использовала автоматическое получение параметров через DHCP, затем была настроена статическая конфигурация: IP-адрес, шлюз по умолчанию и DNS-серверы.

При работе с netplan был получен практический опыт:

- чтения текущих параметров сетевых интерфейсов;
- понимания роли loopback-интерфейса;
- отличия динамической конфигурации от статической;
- настройки адреса, маршрута по умолчанию и DNS;
- применения конфигурации;
- проверки сохранения настроек после перезагрузки.

После настройки сеть проверялась не только по IP-адресу, но и по доменному имени. Это важно, потому что успешный ping до IP подтверждает маршрутизацию, а успешное обращение к домену дополнительно подтверждает работу DNS.

### Обновление системы и пакетный менеджмент

Системные пакеты были обновлены через стандартные инструменты Ubuntu. На практике была закреплена разница между обновлением индекса доступных пакетов и установкой новых версий уже установленных компонентов.

Этот блок дал понимание базового жизненного цикла поддержки Linux-сервера: система должна не только быть установлена, но и регулярно обновляться. Для администратора важно понимать, что список доступных версий и сами установленные пакеты обновляются разными действиями.

### Синхронизация времени

Была настроена автоматическая синхронизация времени. Это важная часть серверной эксплуатации: корректное время влияет на логи, cron, сетевую диагностику, аутентификацию и анализ инцидентов.

Практический вывод: время в системе является не просто отображаемым параметром. Если оно настроено неверно, сложнее анализировать события, сопоставлять журналы разных машин и отлаживать сетевые или сервисные проблемы.

### Работа в консольных редакторах

Была выполнена практика с тремя консольными редакторами: Vim, Nano и Mcedit. В каждом редакторе выполнялись типовые операции:

- создание и редактирование текстового файла;
- сохранение изменений;
- выход без сохранения;
- поиск текста;
- замена текста.

Этот опыт полезен для реального администрирования, потому что на серверах часто нет графической среды, а исправлять конфигурации нужно быстро и уверенно. Особенно важным было не просто открыть файл, а понять разные модели работы редакторов: режимы Vim, сочетания клавиш Nano и меню Mcedit.

### SSHD и удаленный доступ

Был установлен и настроен OpenSSH Server. Служба была запущена, добавлена в автозагрузку и переведена на нестандартный порт. После изменения конфигурации выполнялся перезапуск службы и проверка, что SSH действительно слушает нужный порт.

Дополнительно был разобран случай, когда socket-активация мешает ожидаемому применению порта. Это дало полезный опыт работы не только с конфигурационным файлом, но и с systemd-механизмами, которые могут влиять на поведение сервиса.

Практические результаты:

- установка серверной службы;
- управление службой через systemd;
- изменение параметров SSH;
- проверка процесса через список процессов;
- проверка слушающих TCP-портов через netstat;
- понимание роли `Local Address`, состояния соединения и числового отображения портов.

Для работодателя этот блок показывает базовую готовность работать с удаленным доступом к Linux-серверам и проверять состояние службы не "на глаз", а через процессы, порты и логи.

### Мониторинг процессов и ресурсов

Была выполнена диагностика состояния системы через `top` и `htop`. Анализировались:

- uptime;
- количество пользователей;
- load average;
- количество процессов;
- загрузка CPU;
- использование памяти;
- процессы с наибольшим потреблением CPU и RAM;
- сортировка процессов по PID, CPU, памяти и времени;
- поиск конкретных процессов;
- настройка отображаемых метрик в htop.

Практическая ценность: появилось понимание, как быстро оценивать состояние сервера и отличать разные виды нагрузки. Например, высокий load average, занятая память и высокий CPU требуют разных гипотез и разных следующих проверок.

### Диски, файловые системы и использование пространства

Были использованы инструменты `fdisk`, `df`, `du` и `ncdu`.

Через `fdisk` была получена информация о диске, его размере, секторах и swap. Через `df` проверялись размер, занятое и свободное место на файловой системе, процент использования и тип файловой системы. Через `du` и `ncdu` анализировались размеры каталогов, включая `/home`, `/var` и `/var/log`.

Этот блок дал практическое понимание трех разных уровней диагностики:

- `fdisk` показывает структуру диска и разделов;
- `df` показывает использование смонтированных файловых систем;
- `du` и `ncdu` помогают понять, какие каталоги занимают место.

Это важный навык для эксплуатации: при нехватке места нужно быстро понять, проблема в размере раздела, заполнении конкретной файловой системы или росте отдельных каталогов, например логов.

### Системные журналы

Была выполнена работа с системными логами: просмотр журналов загрузки, системных событий и авторизации. По логам определялись время последнего успешного входа, пользователь и способ входа.

Также выполнялся перезапуск SSH-службы с последующей проверкой события в логах. Это закрепило важный принцип: действие администратора должно быть подтверждено наблюдаемым следом в системе.

Практический результат:

- понимание назначения разных лог-файлов;
- поиск событий авторизации;
- проверка событий служб;
- связь между systemd-операциями и журналами;
- базовый навык расследования состояния системы через логи.

### Cron и автоматизация

Была настроена периодическая задача через cron: выполнение команды по расписанию с последующей проверкой списка задач и записей в системных журналах. После проверки задания были удалены.

Этот блок дал опыт базовой автоматизации и контроля ее выполнения. Важно, что проверялась не только запись в crontab, но и фактическое срабатывание по логам.

Практический вывод: cron полезен для регулярных операций, но его нужно сопровождать проверкой выполнения, корректным расписанием и очисткой ненужных задач.

## Часть 2. Сетевое администрирование Linux

### IPv4-адресация, маски и диапазоны

Сетевой проект начался с расчета адресов и масок. Были разобраны разные формы записи масок: десятичная, префиксная и двоичная. На практике рассчитывались адрес сети, минимальный и максимальный адрес хоста, а также принадлежность адресов к private/public-диапазонам.

Отдельно была разобрана loopback-сеть 127.0.0.0/8: какие адреса относятся к локальному хосту, а какие уже не являются loopback.

Практическая ценность:

- понимание CIDR-записи;
- перевод масок между форматами;
- расчет диапазонов хостов;
- распознавание частных адресных пространств;
- понимание того, что IP-адрес сам по себе недостаточен без маски.

Это фундамент для любой сетевой работы: невозможно корректно настроить маршрут, firewall или DHCP, если нет понимания границ подсети.

### Статическая маршрутизация между двумя машинами

Были подняты две виртуальные машины с разными сетевыми адресами. Для каждой машины была настроена внутренняя сеть через netplan. Затем маршруты между машинами добавлялись двумя способами:

- вручную через команду добавления маршрута;
- постоянно через конфигурационный файл netplan.

Связность проверялась через ping.

Этот блок помог понять разницу между временным изменением состояния ядра и постоянной конфигурацией, которая переживает перезагрузку. Ручной маршрут удобен для быстрой проверки гипотезы, но для устойчивой настройки нужен конфигурационный файл.

### Измерение скорости через iperf3

После настройки связности была измерена пропускная способность между виртуальными машинами. В отчете зафиксированы результаты порядка нескольких гигабит в секунду в обе стороны: примерно 6.49 Gbit/s и 6.82 Gbit/s.

Также выполнялся перевод единиц скорости: Mbps, MB/s, Kbps, Gbps. Это важно, потому что в сетях часто смешиваются биты и байты, а ошибка в единицах может привести к неправильной оценке производительности.

Практический результат:

- запуск iperf3 в роли сервера и клиента;
- измерение пропускной способности в обоих направлениях;
- понимание различий между сетевыми и файловыми единицами измерения;
- получение базового опыта performance-проверки сети.

### Firewall на iptables

Была создана простая firewall-конфигурация через iptables. На двух машинах проверялись разные порядки правил: в одном случае раньше срабатывало запрещающее правило, в другом - разрешающее. Это позволило на практике увидеть, что порядок правил критичен: обработка идет сверху вниз до первого подходящего совпадения.

Проверялась блокировка и разрешение ICMP-ответов, а также доступность портов SSH и HTTP. После этого через ping определялась машина, которая не отвечает на ICMP, а через nmap подтверждалось, что хост при этом остается включенным.

Практическая ценность:

- понимание цепочек и правил iptables;
- влияние порядка правил на результат;
- различие между "хост не отвечает на ping" и "хост недоступен";
- использование nmap как дополнительного инструмента проверки;
- базовое понимание фильтрации ICMP/TCP-трафика.

Это важный опыт, потому что в реальных системах отказ ping не всегда означает проблему с сервером. Иногда это следствие политики firewall.

### Построение сети с маршрутизаторами и рабочими станциями

В более сложной части была настроена сеть из нескольких рабочих станций и маршрутизаторов. Машины получили адреса в разных подсетях, маршрутизаторы соединяли эти сети между собой, а на рабочих станциях задавались шлюзы по умолчанию.

На маршрутизаторах включалась пересылка IP-пакетов. Настройка выполнялась как временно, так и постоянно через системную конфигурацию.

Практический результат:

- настройка нескольких сетевых интерфейсов;
- назначение адресов в разных подсетях;
- настройка default route на рабочих станциях;
- добавление статических маршрутов на маршрутизаторах;
- включение IP forwarding;
- проверка таблиц маршрутизации через `ip r`;
- проверка прохождения трафика между подсетями.

Этот блок показал, как Linux-машина может работать не только как конечный хост, но и как маршрутизатор.

### Принцип выбора маршрута

Отдельно был разобран выбор маршрута ядром Linux. На практике сравнивались более конкретный маршрут до определенной сети и маршрут по умолчанию. Был закреплен принцип longest prefix match: если адрес назначения подходит под несколько маршрутов, выбирается наиболее специфичный.

Практический вывод: default route является запасным вариантом, а не универсальным маршрутом для всего трафика. Если в таблице есть более точная запись, ядро выберет ее.

Это понимание важно при отладке сетей: неправильная или слишком широкая route-запись может перенаправить трафик не туда, а отсутствие специфичного маршрута приведет к использованию default gateway.

### Traceroute, tcpdump и ICMP

Для анализа пути пакетов использовались traceroute/tracepath и tcpdump. На маршрутизаторе перехватывался трафик, проходящий через конкретный интерфейс, а затем анализировалось, как пакеты проходят по сети.

Также проверялась ситуация обращения к несуществующему адресу. Через ICMP было видно, что сеть сообщает об ошибках доставки, например о недоступности направления.

Практическая ценность:

- понимание роли TTL при построении маршрута;
- наблюдение трафика на интерфейсе через tcpdump;
- анализ ICMP как протокола служебных сообщений;
- связь между маршрутизацией и диагностическими утилитами;
- навык проверки не только конечного результата, но и промежуточного прохождения пакетов.

Этот опыт особенно полезен для troubleshooting: если ping не проходит, важно уметь понять, где именно теряется пакет.

### DHCP

Была настроена динамическая выдача IP-адресов через DHCP-сервер. На маршрутизаторе задавались параметры сети, шлюза по умолчанию и DNS. После перезапуска службы клиентская машина получала адрес автоматически, а связность проверялась ping.

Дополнительно был настроен сценарий с привязкой адреса к MAC-адресу. Это позволило сравнить обычную динамическую выдачу адресов и фиксированную выдачу для конкретного клиента.

Также выполнялось обновление DHCP-аренды. После обновления адрес остался прежним, что соответствует нормальному поведению: сервер продлевает текущую аренду для того же клиента, если адрес доступен и привязка не изменилась.

Практический результат:

- установка и настройка DHCP-сервера;
- задание параметров сети, DNS и шлюза;
- перезапуск и проверка службы;
- получение адреса клиентом;
- настройка fixed-address по MAC;
- обновление DHCP lease;
- понимание жизненного цикла аренды адреса.

### NAT, SNAT и DNAT

В завершающей части сетевого проекта был настроен доступ между сегментами через NAT. Для проверки использовался Apache2, настроенный на прослушивание внешнего интерфейса, и telnet для тестирования TCP-соединений.

Сначала firewall на маршрутизаторе блокировал пересылку пакетов, затем отдельно разрешался ICMP, а после этого добавлялись правила NAT:

- SNAT для подмены адреса источника;
- DNAT для перенаправления входящего соединения на внутренний хост.

Практический результат:

- запуск и проверка HTTP-службы;
- настройка политики FORWARD;
- разрешение ICMP для диагностики;
- настройка трансляции адресов;
- проверка TCP-доступности через telnet;
- понимание различия между маршрутизацией и NAT.

Этот блок дал практическое представление о том, как внутренние сервисы могут быть доступны через адрес маршрутизатора и как исходящий трафик может скрываться за внешним адресом.

## Что было изучено на практике

### Работа с Linux как серверной системой

Был получен опыт работы с Linux не как с пользовательской ОС, а как с серверной платформой. Это включает управление пользователями, службами, сетью, пакетами, логами, дисками и задачами по расписанию.

Главный результат - понимание, что Linux-администрирование строится на наблюдаемости: любое изменение нужно подтверждать командой, логом, состоянием службы или сетевым тестом.

### Связь между конфигурацией и состоянием системы

На практике стало понятно различие между:

- временными изменениями, которые действуют до перезагрузки;
- постоянными настройками в конфигурационных файлах;
- фактическим состоянием, которое показывает ядро или служба в данный момент.

Это проявлялось в netplan, маршрутах, IP forwarding, DHCP, SSH и iptables.

### Диагностика вместо угадывания

Практика научила проверять гипотезы инструментами. Если сеть не работает, проверяются интерфейсы, адреса, маршруты, firewall, DNS, доступность хоста, путь пакета и логи. Если служба не работает, проверяются systemd-статус, процесс, порт и записи в журналах.

Такой подход ближе к реальной эксплуатации, где важно не просто "перезапустить", а понять причину.

### Понимание сетевого стека

В сетевом проекте удалось связать теорию IPv4 с практикой:

- IP-адрес и маска определяют принадлежность к сети;
- маршрутизация решает, куда отправить пакет;
- firewall может разрешить или заблокировать трафик;
- ICMP помогает диагностировать ошибки;
- DHCP автоматизирует выдачу параметров;
- NAT меняет адреса и позволяет строить доступ между сетевыми зонами.
